# 专项优化：缓存失效版本管理

## 1. 背景

在完成统一页面缓存容器之后，缓存虽然已经收口，但失效规则仍然主要依赖：

- 手动清理某个 scope
- 重新覆盖缓存
- 使用对象引用或 `identityHashCode` 间接区分数据变化

这种方式在短期内可用，但随着页面缓存继续增多，会逐步出现“该失效时未失效”或“为了安全只能整块清空”的问题。

## 2. 当前问题

当前缓存失效方式存在以下不足：

- 失效条件分散在页面内部，不够统一
- 某些页面依赖对象引用变化来判定缓存是否可复用
- 数据写入后，并没有一套统一的版本推进机制
- 后续继续扩展缓存时，很容易重复引入不同失效写法

## 3. 本次优化目标

本次目标是给缓存体系补上一层“统一版本号管理”，让页面缓存 key 不再依赖临时引用，而是显式依赖数据版本。

目标如下：

- 提供共享的数据版本管理工具
- 为歌曲数据相关缓存引入统一版本号
- 在明确的数据写入点推进版本号
- 让页面缓存 key 能稳定表达“当前数据版本”

## 4. 设计方案

本次新增一个轻量级版本管理器，例如：

- `song_library`
- 后续也可以扩展 `playlists`
- 未来还可以扩展 `settings_snapshot`、`stats` 等

页面缓存 key 统一改为包含版本号，例如：

- `song_library:v12|sort=name|asc=1`

这样当歌曲数据写入、删除、重扫后，只要推进 `song_library` 版本，就会自然命中新的缓存 key，而无需页面自己猜测数据是否变化。

## 5. 首批落地范围

本次优先接入歌曲数据版本，因为它影响面最大：

- `SongsPage` 可见列表缓存
- `AlbumsPage` 分组缓存
- `ArtistsPage` 分组缓存
- `HomePage` 统计缓存

这些页面都直接依赖歌曲表或歌曲聚合结果。

## 6. 版本推进时机

本次先在明确且安全的写入点推进版本：

- `SongDao.upsertSongs(...)`
- `SongDao.deleteByIds(...)`
- `SongDao.deleteBySource(...)`

这样做的好处是：

- 歌曲库数据发生变化时，统一从底层推进版本
- 页面不需要知道是谁改了数据
- 版本提升后，新的缓存 key 自然生效

## 7. 预期收益

- 页面缓存失效规则更稳定
- 减少依赖对象引用或手工清空 scope 的写法
- 后续新页面只要依赖版本号即可加入统一缓存体系
- 为更完善的缓存策略打基础

## 8. 后续可继续扩展的方向

本次完成后，后续建议继续扩展：

- 为歌单相关缓存引入 `playlists` 版本号
- 给版本管理器增加持久化选项（若未来需要跨进程缓存控制）
- 为缓存命中调试增加版本可视化

## 9. 验收标准

本次优化完成后应满足：

- 新增共享版本管理工具
- 歌曲库写入后会自动推进版本
- 高优先级歌曲相关页面缓存 key 使用版本号
- `flutter analyze` 通过
- 基础测试通过
