# 专项优化 - SongsPage 动作控制层拆分

## 背景

虽然 `SongsPage` 已经拆出了可见列表控制、封面协调和多选控制，但页面内部仍然残留一大块“动作型逻辑”，包括：

- 批量加歌单
- 批量移除歌曲
- 批量刮削元数据与歌词
- 刮削进度浮层
- 删除过程的缓存清理与播放器联动

这部分逻辑有一个共同特征：

- 都不是页面布局本身
- 都带有多步异步流程
- 都会修改数据库、缓存、播放队列或页面状态

如果继续堆在 `songs_page.dart`，后续再加下载、批量收藏、批量导出等动作时，页面会继续膨胀。

## 本次目标

1. 将 `SongsPage` 中的动作逻辑提取到独立控制层。
2. 页面只负责提供上下文、状态读写入口和 UI 绑定。
3. 保持当前交互行为不变，不做大范围重构。

## 拆分范围

本次优先收口以下动作：

- 批量加歌单
- 批量移除歌曲
- 批量刮削
- 歌曲删除后的缓存清理

暂不处理：

- 页面列表渲染
- 封面预取
- 多选状态本身

## 实施方式

### A. 新增 `SongsActionsController`

控制层负责：

- 解析歌曲请求头
- 判断是否跳过标签刮削
- 单曲刮削
- 批量刮削任务编排
- 批量移除与缓存清理
- 批量加歌单入口收口

### B. 页面层只保留 UI 宿主职责

`SongsPage` 保留：

- overlay 和 dialog 的展示层
- signal / notifier 状态容器
- controller 回调绑定

这样能避免把 `BuildContext`、页面 signal 和底层 service 绑死在控制层里。

## 预期收益

- `SongsPage` 继续瘦身，职责更清晰
- 动作逻辑更集中，便于后续继续拆批量功能
- 后面若要把动作迁移到别的页面或复用，也更容易

## 验证建议

1. 多选歌曲后加入歌单，确认行为不变。
2. 多选歌曲后移除，确认数据库、页面列表、缓存和播放队列都同步更新。
3. 执行批量刮削，确认进度展示与最终更新正常。
