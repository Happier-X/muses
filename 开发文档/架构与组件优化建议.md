# 架构与组件优化建议

本文档基于当前 `nagomusic` Flutter 工程的页面结构、状态组织、组件拆分和近期性能问题排查结果整理，目标是给后续迭代提供一份可执行的优化清单。

文档重点覆盖四类问题：

- 页面结构与职责边界
- 组件复用与可维护性
- 数据流与状态管理一致性
- 首屏、切页、列表渲染的性能优化方向

## 1. 当前项目的主要特点

从现有代码看，项目已经形成了较完整的业务结构：

- 页面主要集中在 `lib/pages/`
- 服务层主要集中在 `lib/app/services/`
- 通用组件主要集中在 `lib/components/`
- 页面普遍采用 `signals` / `ValueNotifier` / service singleton 的混合模式
- 页面经常直接调用 DAO、service、弹窗、导航和本地缓存逻辑

当前结构能支持功能快速迭代，但随着功能增加，部分页面已经开始承担过多职责，出现以下问题：

- 页面文件过大，阅读和修改成本提升
- 缓存策略分散，缺少统一约束
- 页面初始化逻辑重复，容易出现体验不一致
- 某些组件既做 UI 又做数据/缓存/并发控制，职责过重
- 切页、首屏和大列表渲染时，容易发生瞬时性能抖动

## 2. 当前最值得优先优化的区域

### 2.1 大页面职责过重

以下页面已经明显承担了过多逻辑：

- `lib/pages/songs/songs_page.dart`
- `lib/pages/library/playlists_page.dart`
- `lib/pages/source/source_page.dart`
- `lib/pages/home/home_page.dart`

这些页面通常同时承担：

- 读取本地偏好设置
- 请求数据库或 service
- 进行分组、排序、过滤
- 管理弹窗、底部菜单、确认框
- 处理路由跳转
- 维护页面局部缓存
- 构建完整 UI

这会带来两个直接问题：

- 任何一项性能问题都更难定位
- 新需求进入后容易继续堆在页面里，形成“超级页面”

### 2.2 页面初始化逻辑不统一

当前多个页面都有自己的 `initState -> _init -> _load` 逻辑链，但每个页面的节奏不完全一致。

例如：

- 有些页面在 `initState` 中直接重计算
- 有些页面已经改成首帧后延迟初始化
- 有些页面进入时优先显示缓存，有些页面则仍然等待完整计算

这会造成体验差异：

- 某些页面进入很顺
- 某些页面进入仍然会明显掉帧
- 某些页面返回后状态保留较好，某些则会重新初始化

### 2.3 缓存策略分散

项目已经开始引入多种缓存，但目前缺少统一规范：

- `SongsPage` 有可见列表缓存和封面缓存
- `AlbumsPage` / `ArtistsPage` 有分组结果缓存
- `HomePage` 有统计缓存
- `ArtworkWidget` 又有独立的图片字节缓存与加载队列

这些缓存是有收益的，但如果没有统一规则，会逐步出现：

- 失效条件不一致
- 相同数据在不同页面被重复缓存
- 页面间行为不一致
- 优化点难以复用

### 2.4 组件层职责偏重

`ArtworkWidget` 是当前最典型的例子。这个组件不仅负责展示，还负责：

- 文件存在判断
- 原图/缓存图选择
- 元数据读取
- 图片压缩
- 加载并发控制
- LRU 缓存
- 任务队列调度

这类组件虽然短期内好用，但长期会造成：

- UI 组件难测试
- 页面侧和组件侧出现重复逻辑
- 性能问题更难集中治理

## 3. 建议的目标架构方向

不建议一次性大重构，而是按“低风险、可渐进迁移”的方式推进。

推荐逐步形成以下分层：

### 3.1 页面层 Page

页面层只负责：

- 页面布局
- 交互入口绑定
- 基于状态渲染 UI
- 调用 controller / service 的公开方法

页面层应尽量避免直接做：

- 大规模数据计算
- 多步异步编排
- 缓存失效策略判断
- 复杂数据转换

### 3.2 页面控制层 Controller / ViewModel

这一层负责：

- 页面初始化流程
- 读取和保存 prefs
- 加载 service / DAO 数据
- 组装 UI 需要的展示态
- 维护页面级缓存
- 对外暴露 signal / notifier

适合迁移到这一层的对象包括：

- `SongsPage` 的列表构建控制
- `HomePage` 的统计加载控制
- `SourcePage` 的扫描状态控制
- `PlaylistsPage` 的排序和管理动作控制

### 3.3 服务层 Service / ReadModel

服务层除了现有 service 外，建议增加“只读聚合服务”。

例如：

- `LibraryOverviewService`
- `AlbumReadService`
- `ArtistReadService`
- `ArtworkService`

它们负责把底层 DAO/service 数据整合成页面真正需要的读模型，而不是让页面自己拼。

### 3.4 展示组件层 Widgets

组件层应尽量专注于：

- 纯展示
- 轻交互
- 接收数据并渲染

避免在组件层塞入：

- 重异步任务
- 队列调度
- 持久缓存策略
- 多来源数据装配

## 4. 页面结构优化建议

### 4.1 `SongsPage` 拆分优先级最高

`lib/pages/songs/songs_page.dart` 当前承担了最多逻辑，建议优先拆分。

推荐拆分方向：

- `songs_page.dart`：页面外壳和主要布局
- `songs_page_controller.dart`：筛选、排序、可见列表、缓存协调
- `songs_artwork_coordinator.dart`：封面预取、任务队列、缓存协调
- `songs_selection_controller.dart`：多选逻辑和底部操作
- `songs_actions.dart`：删除、歌词抓取、更新等动作逻辑

这样可以直接获得几个好处：

- 后续再调性能时不需要反复翻一个超大文件
- 列表和封面问题可以分开处理
- 多选/歌词/删除等行为更容易独立测试

### 4.2 `PlaylistsPage` 适合拆动作层

`lib/pages/library/playlists_page.dart` 当前同时管理：

- 歌单加载
- 排序
- 拖拽重排
- 新建/重命名/删除
- 弹窗与详情跳转

建议拆成：

- `playlists_page.dart`
- `playlists_controller.dart`
- `playlist_actions.dart`
- `playlist_sorting.dart`

### 4.3 `SourcePage` 适合拆扫描控制层

`lib/pages/source/source_page.dart` 里本地扫描和 WebDAV 扫描状态集中在页面内部。

建议抽出：

- `source_scan_controller.dart`
- `source_data_controller.dart`

这样后续如果要支持：

- 后台扫描
- 扫描中断恢复
- 统一进度通知

都更容易扩展。

### 4.4 `HomePage` 应保持轻量

首页本质上应该是一个轻量入口页，不应继续堆积数据计算责任。

建议把首页统计逻辑提炼为：

- `library_overview_controller.dart`

页面只负责：

- 显示统计卡片
- 显示入口卡片
- 触发跳转

## 5. 组件层优化建议

### 5.1 `ArtworkWidget` 应做职责收口

当前 `lib/components/common/artwork_widget.dart` 职责过重。

建议目标：

- `ArtworkWidget` 只负责展示和加载状态显示
- 把元数据读取、压缩、并发、缓存等逻辑下沉到 `ArtworkService`

推荐拆法：

- `artwork_widget.dart`：只管渲染
- `artwork_service.dart`：负责读取、压缩、缓存
- `artwork_request_queue.dart`：负责并发和优先级

这样可以和 `SongsPage` 自己的封面逻辑合并，减少重复实现。

### 5.2 `AppPageScaffold` 已接近超级容器

`lib/components/layout/base/app_page_scaffold.dart` 目前同时负责：

- safe area
- mini player
- bottom navigation
- drawer
- keyboard inset
- tablet 模式分支

这是一个典型“功能已经够多”的基础容器。

建议拆分：

- `app_page_scaffold.dart`：只保留基础 page scaffold
- `drawer_shell.dart`：处理抽屉和页面位移
- `bottom_overlay_host.dart`：处理 mini player / nav overlay

这样后续改动动画和布局时会安全很多。

### 5.3 `SideMenu` 导航语义需要统一

`lib/components/layout/side_menu.dart` 中既支持回调注入，又有默认导航行为。

建议统一一套导航约定：

- 库类页面是否都走 push
- 某些主页面是否走 replace
- 设置/详情页是否总是 push

如果不统一，未来非常容易出现：

- 从首页进入和从侧边栏进入行为不同
- 页面返回栈体验不一致
- 动画和状态保留策略不一致

### 5.4 组件导出策略可以更克制

`lib/components/index.dart` 当前承担大而全的 barrel export。

建议：

- 保留常用稳定组件导出
- 对复杂页面优先直接引入具体组件文件

这样能让依赖更清晰，也方便后续删除/重构组件。

## 6. 状态管理与数据流建议

### 6.1 明确 `signals`、`ValueNotifier`、service state 的边界

当前项目混合使用是可行的，但建议形成约定：

- 页面即时 UI 状态：优先 `signal`
- 页面控制器的轻量可观察状态：可继续用 `signal`
- 需要跨页面共享的简单状态：放到 state/service 中
- 进度型或外部 listener 型状态：保留 `ValueNotifier`

关键目标不是统一成一种工具，而是减少“同一类问题用三种方式”的不确定性。

### 6.2 建立统一缓存失效规则

建议明确三类缓存：

- 数据结果缓存：如歌曲分组、专辑分组、首页统计
- 资源缓存：如封面图、压缩结果
- 视图状态缓存：如滚动位置、当前筛选、当前排序

每类缓存都应该定义：

- 缓存 key 如何生成
- 什么时候失效
- 最大容量是多少
- 命中时如何复用

当前建议优先约定：

- 数据版本变化时失效
- prefs 变化时按 key 局部失效
- 页面返回时优先复用，不主动重算

### 6.3 逐步引入读模型服务

目前很多页面都直接拿 `SongDao.fetchAllCached()` 然后自己做二次处理。

建议逐步抽出：

- `LibraryOverviewReadService`
- `SongsReadService`
- `AlbumsReadService`
- `ArtistsReadService`

这样页面只拿“可以直接展示的数据”，而不是都从歌曲原始表开始拼装。

## 7. 性能优化方向

### 7.1 首屏与切页性能

近期已经排查出，切页卡顿主要来自：

- 页面初始化重计算
- 封面加载与压缩并发过多
- 返回首页时统计重新拉取
- 列表页面每次进入重新分组或重排

后续继续优化时，建议坚持以下原则：

- 首帧优先显示可用 UI，不等待完整计算
- 大计算放到首帧后执行
- 可见区优先，非可见区延后
- 缓存命中优先，不重复做全量计算

### 7.2 列表页保留状态

建议统一约定：

- 列表页返回后保留滚动位置
- 排序和筛选条件保留
- 分组和列表快照尽量保留

这类优化比单纯“再快一点”更能提升实际体感。

### 7.3 首屏图片策略继续收敛

建议把封面相关策略统一：

- 限制并发数量
- 首帧后再预取
- 只预取可见区附近
- 本地有 `localCoverPath` 时不重复走元数据读取链

同时应避免页面自己一套、组件自己一套的图片加载策略长期共存。

## 8. 推荐的迭代顺序

为了避免大范围重构带来风险，建议按以下顺序推进：

### 第一阶段：统一页面初始化和缓存规范

- 抽公共页面初始化 mixin 或 controller 模板
- 整理首页、歌曲、专辑、艺术家页面的缓存策略
- 统一页面切入时的加载节奏

### 第二阶段：收口封面能力

- 抽 `ArtworkService`
- 合并 `SongsPage` 和 `ArtworkWidget` 的封面队列/缓存逻辑
- 统一并发与压缩策略

### 第三阶段：拆大页面

- 先拆 `SongsPage`
- 再拆 `SourcePage`
- 再拆 `PlaylistsPage`

### 第四阶段：整理基础容器与导航

- 精简 `AppPageScaffold`
- 明确 `SideMenu` 的导航协议
- 统一主页面和详情页的路由策略

### 第五阶段：引入读模型服务

- 把常见页面从“直接拿 SongDao”逐步迁移到 read service

## 9. 当前最推荐优先落地的五项

如果只做最有价值的五项，建议按这个顺序：

1. 抽统一页面初始化模式，避免每页各自管理首帧和加载节奏
2. 抽 `ArtworkService`，统一封面读取、压缩、缓存和队列
3. 拆 `SongsPage`，优先拆控制层和动作层
4. 给首页/专辑/艺术家建立一致的缓存失效规则
5. 精简 `AppPageScaffold` 和统一 `SideMenu` 导航语义

## 10. 总结

当前项目的基础结构并不混乱，已经具备明确的页面、服务和组件分层，只是随着功能增加，几个核心页面和基础组件开始承担过多职责。

下一阶段最重要的目标不是“大改架构”，而是：

- 让页面职责更单一
- 让缓存和初始化策略更一致
- 让性能优化点集中到少数服务/控制层
- 让组件更多回到展示职责

只要沿着这个方向渐进优化，后续新增功能和继续做性能优化都会轻松很多。
