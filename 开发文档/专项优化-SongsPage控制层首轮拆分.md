# 专项优化：SongsPage 控制层首轮拆分

## 1. 背景

`SongsPage` 是当前项目中职责最重的页面之一，已经同时承担：

- 页面初始化
- 偏好读取与保存
- 歌曲列表加载
- 筛选与排序
- 可见列表缓存
- 封面预取
- 多选与批量操作
- 歌词抓取与删除

虽然前面已经完成了页面初始化统一、封面能力收口和缓存统一，但 `SongsPage` 本身仍然过大，后续继续维护和优化成本依旧偏高。

## 2. 本次目标

本次不做大拆分，而是先完成“首轮控制层提取”，聚焦收益最高、风险较低的一部分：

- 把可见歌曲列表构建逻辑从页面中抽离
- 把筛选、排序、首批快显、异步精排收口到独立 controller
- 页面只负责接收 controller 结果并更新 UI 状态

## 3. 本次拆分范围

本次只拆以下职责：

- 快速筛选 `seedVisibleSongsFast`
- 完整可见列表构建
- 排序与筛选 isolate 调用
- 版本化缓存 key 生成与复用

暂不处理的职责：

- 封面预取
- 多选
- 删除
- 歌词抓取
- 播放队列联动

## 4. 设计方案

新增一个轻量控制器，例如 `SongsVisibleController`，负责：

- 基于 `songs/sourceFilter/sortKey/ascending` 生成结果
- 管理页面缓存容器的读写
- 输出 `allVisible` 和 `displayVisible`
- 提供给页面的首屏快显入口

页面只需要：

- 提供当前输入条件
- 接收 controller 返回结果
- 再决定滚动窗口、封面预取和当前播放状态同步

## 5. 预期收益

- `SongsPage` 页面代码进一步瘦身
- 可见列表构建逻辑从页面职责中移出
- 后续继续拆控制层时，有明确起点
- 对筛选/排序性能问题的排查入口更集中

## 6. 验收标准

- 新增 `SongsVisibleController`
- `SongsPage` 不再直接持有主要的可见列表构建实现
- 原有列表行为保持一致
- `flutter analyze` 通过
- 基础测试通过
