# 专项优化 - 启动自动刷新音源

## 背景

当前本地音源与 WebDAV 音源都支持手动扫描，但进入应用后不会自动检查新增歌曲。用户希望应用在启动时可以自动刷新音源，并且本地与云端分别配置。

这类需求的关键点不是“简单进应用就全量扫描”，而是：

- 不阻塞首页首屏显示
- 本地、云端开关独立
- 尽量复用现有扫描实现
- 新歌能够自动写入媒体库

## 本次目标

1. 在设置中新增两个独立开关：
   - 启动时自动刷新本地音源
   - 启动时自动刷新云端音源
2. 应用进入首页后，后台异步触发刷新，不阻塞首屏
3. 刷新成功后自动把新增歌曲写入现有媒体库
4. 保持现有手动扫描入口不变

## 设计原则

### 1. 启动刷新不能影响首页首帧

- 首页先正常加载统计和内容
- 启动刷新使用延后触发的后台任务
- 不弹扫描进度对话框，不打断用户操作

### 2. 本地和云端分别控制

- 本地音源可能依赖系统媒体权限
- 云端刷新可能依赖网络和远程目录性能
- 两者要独立开关，互不绑定

### 3. 优先复用现有扫描逻辑

- 本地继续使用 `LocalMusicService.scan(...)`
- 云端继续使用 `WebDavMusicService.scan(...)`
- 启动自动刷新只是在时机和 UI 上做“静默模式”封装

## 实施方案

### A. 新增启动刷新设置

- 在全局设置状态里增加：
  - `autoRefreshLocalOnLaunch`
  - `autoRefreshCloudOnLaunch`
- 默认关闭，避免无感增加启动负担

### B. 提供统一启动刷新服务

- 新增独立服务负责启动后的静默刷新调度
- 服务负责：
  - 加载设置
  - 检查本地权限 / 云端配置
  - 分别触发本地与 WebDAV 扫描
  - 避免同一次启动重复执行

### C. 首页启动时异步触发

- 在 `HomePage` 中延迟触发
- 不显示扫描弹窗
- 允许失败静默降级，必要时仅在 debug 输出日志

## 风险与注意点

- 本地音源未授权时，不能反复打断式申请权限
- WebDAV 源较多时，启动刷新可能消耗时间较长
- 启动刷新完成后，首页统计可能需要重新刷新才能反映新增数量

## 第一阶段范围

1. 设置中新增本地 / 云端独立启动刷新开关
2. 首页进入后后台静默执行
3. 刷新完成后自动更新媒体库和首页统计

## 验证建议

1. 开启“启动时刷新本地音源”，放入新本地歌曲后重启应用，确认自动入库
2. 开启“启动时刷新云端音源”，WebDAV 目录新增歌曲后重启应用，确认自动入库
3. 关闭任一开关后，对应来源不再自动刷新
4. 首屏进入时确认不会弹扫描对话框，也不会明显卡顿
