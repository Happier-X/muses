# 专项优化：歌单缓存版本管理

## 1. 背景

在歌曲库缓存完成版本化失效之后，歌单相关数据仍然缺少同等级别的统一版本控制。

歌单页面和歌单选择场景都存在这样的问题：

- 歌单创建、重命名、删除后，需要重新读取
- 歌单排序或歌曲增删后，旧缓存可能继续被命中
- 页面之间如果后续引入歌单缓存，没有统一版本很容易出现不一致

## 2. 本次目标

本次优化聚焦两个点：

- 为歌单数据引入统一版本号 `playlists`
- 让歌单页面缓存 key 明确依赖这个版本号

## 3. 设计方案

- 在 `PlaylistsService` 中定义歌单缓存版本域
- 在所有明确的歌单写操作后推进版本号
- 在 `PlaylistsPage` 中引入统一缓存容器，并将 key 改成 `playlistsv:<version>` 风格

## 4. 首批纳入版本推进的写操作

- 创建歌单
- 重命名歌单
- 删除歌单
- 向歌单添加歌曲
- 从歌单移除歌曲
- 歌单排序调整
- 歌单内歌曲排序调整
- 置顶歌单

## 5. 预期收益

- 歌单页不需要依赖手动猜测何时清缓存
- 未来歌单相关页面、弹窗、选择器都能复用同一版本规则
- 页面来回切换时可以安全使用缓存，同时保证数据更新后自然失效

## 6. 验收标准

- `PlaylistsService` 具备统一版本推进
- `PlaylistsPage` 缓存 key 依赖歌单版本号
- 歌单创建/删除/改名/排序后，页面能命中新版本缓存
- `flutter analyze` 通过
- 基础测试通过
