# 专项优化：封面加载职责收口

## 1. 背景

当前项目中的封面加载逻辑同时存在于多个位置，主要包括：

- `lib/components/common/artwork_widget.dart`
- `lib/pages/songs/songs_page.dart`
- `lib/app/services/artwork_cache_helper.dart`

其中 `ArtworkWidget` 既负责 UI 展示，也负责：

- 读取本地封面文件
- 读取音频 metadata 中的内嵌图片
- 压缩图片
- 管理并发队列
- 管理内存缓存

而 `SongsPage` 中又维护了一套相似的封面预取和缓存逻辑。这意味着同一类资源的处理规则分散在页面和组件中，长期会带来性能、维护和一致性问题。

## 2. 当前问题

当前封面加载体系存在几个典型问题：

- UI 组件承担了过多非展示职责
- 页面和组件各自维护缓存与并发策略，存在重复实现
- 不同入口的封面行为不完全一致
- 后续要继续优化封面链路时，需要同时改多个地方
- 问题排查成本高，难以判断是页面逻辑还是组件逻辑导致的抖动

## 3. 本次优化目标

本次优化不尝试一次性删除所有页面侧封面逻辑，而是先完成“核心能力收口”的第一步：

- 提供统一的封面读取服务
- 将 `ArtworkWidget` 的重逻辑下沉到 service
- 让组件层尽量回归为展示层
- 为后续继续合并 `SongsPage` 的封面逻辑打基础

## 4. 设计方案

本次计划新增一个共享的 `ArtworkService`，负责以下能力：

- 读取本地缓存封面
- 读取音频 metadata 中的图片
- 对非原图请求进行压缩
- 管理内存缓存
- 管理正在进行中的加载任务
- 限制并发数量

组件和页面通过 service 获取图片字节，而不再自行维护完整加载链。

## 5. 首批落地范围

本次优先改造：

- `lib/components/common/artwork_widget.dart`

保留但暂不完全合并的部分：

- `SongsPage` 中与列表预取强相关的局部逻辑

原因是 `SongsPage` 除了封面读取，还有列表预取窗口、页面缓存、异步更新数据库封面路径等页面级行为。本次先把底层“如何读取与缓存封面”统一，再决定第二步如何让页面预取也完全接入同一套 service。

## 6. 预期收益

- `ArtworkWidget` 代码明显简化
- 统一封面读取、压缩、缓存和并发策略
- 后续继续优化 `SongsPage` 封面链路时可以复用 service
- 降低 UI 组件的复杂度和维护成本

## 7. 后续阶段建议

本次完成后，后续建议继续推进：

- 让 `SongsPage` 预取也直接基于 `ArtworkService`
- 统一不同页面的封面并发上限
- 为 service 增加更明确的缓存失效策略
- 统一“优先本地缓存封面、其次 metadata 封面”的规则

## 8. 验收标准

本次优化完成后应满足：

- `ArtworkWidget` 不再自己维护完整的图片任务队列和缓存逻辑
- 新增统一 `ArtworkService`
- 原有封面显示行为不被破坏
- `flutter analyze` 通过
- 基础测试通过
